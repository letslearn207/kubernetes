01

What is CNI? — The Concept

theory

Before touching any command, understand why CNI exists and what problem it solves.

ANALOGYThe Apartment Building Intercom Problem

Imagine a giant apartment building (your Kubernetes cluster). Each apartment is a Pod. Residents (containers) live inside. Now, for any two residents to call each other, the building needs an intercom system — wiring, phone numbers, routing rules.

CNI is that intercom system. It is a specification that tells Kubernetes: "here is how to wire up networking whenever a new apartment (Pod) is created or destroyed." Kubernetes itself doesn't care how the wiring works — it just calls a standard interface, and the CNI plugin does the actual wiring.

FORMALCNI — Container Network Interface

CNI is a specification + set of libraries (originally from CoreOS, now a CNCF project) that defines:

— A standard way for container runtimes (containerd, CRI-O) to request network setup
— A standard way for plugins to respond with network info (IP address, routes, DNS)
— A plugin model where different network implementations can be swapped in

When Kubernetes needs to start a Pod, it calls the CNI plugin's binary with a JSON config. The plugin sets up the network interface (veth pair, IP, routes) and reports back. When the Pod dies, the plugin cleans everything up.

💡
Key insight: Kubernetes itself has zero networking code. It defines four rules (the Kubernetes network model) and delegates all implementation to CNI plugins. This is why you can swap Flannel for Calico for Weave without changing Kubernetes itself.

// The 4 Kubernetes Network Rules (CNI must satisfy ALL of these)

1. Pod-to-Pod  — Every pod can talk to every other pod across nodes without NAT.
2. Node-to-Pod  — Every node can talk to every pod without NAT.
3. Pod sees its own IP  — The IP a pod sees for itself is the same IP others use to reach it.
4. Services  — A stable virtual IP (ClusterIP) load-balances across pod endpoints (handled by kube-proxy, not CNI).

How CNI Works in Kubernetes

theory

Trace the exact journey from kubectl apply to a Pod getting an IP address.

Step-by-step: Pod creation → IP assignment

When you run kubectl apply -f pod.yaml, here is what happens under the hood:

kubectl apply
→
kube-apiserver
→
kube-scheduler
→
kubelet (on node)
→
CNI Plugin
→
Pod gets IP
🔍
kubelet is the trigger. Once the scheduler assigns the Pod to a node, the kubelet on that node:
1. Asks the container runtime (containerd) to create a pause container (the network namespace holder)
2. Calls the CNI binary: /opt/cni/bin/<plugin> with ADD command
3. Passes a JSON config from /etc/cni/net.d/
4. CNI plugin creates a veth pair, assigns IP, sets routes, and returns the IP to kubelet
5. kubelet stores the IP and reports it to the API server
Key CNI concepts you must know

// veth pair

A virtual Ethernet cable with two ends. One end goes inside the Pod's network namespace, the other stays on the host. Like plugging a LAN cable from a room into the building's switch. Traffic goes in one end and comes out the other.

// Network Namespace

A completely isolated network stack (interfaces, routes, iptables). Every Pod gets its own. Containers in the same Pod share one namespace — that's why they can talk on localhost but are isolated from other pods.

// CNI Config (/etc/cni/net.d/)

JSON files that tell kubelet which CNI plugin to call and with what settings. Example: 10-aws.conflist for AWS VPC CNI. Kubelet reads the lexicographically first file in this directory.

// CNI Binaries (/opt/cni/bin/)

The actual executables: aws-cni, calico, flannel, loopback, portmap, etc. These are called by kubelet and perform the actual kernel-level network setup using Linux netlink calls.

What happens at the Linux kernel level

The CNI plugin ultimately makes these kernel calls to wire up a Pod's network:

bash (conceptual — what CNI does internally)
# 1. Create a veth pair
ip link add veth0 type veth peer name eth0

# 2. Move one end into the Pod's network namespace
ip link set eth0 netns <pod-netns>

# 3. Assign IP address to the pod's interface
ip netns exec <pod-netns> ip addr add 192.168.1.5/24 dev eth0

# 4. Bring up the interfaces
ip netns exec <pod-netns> ip link set eth0 up
ip link set veth0 up

# 5. Add default route inside the Pod
ip netns exec <pod-netns> ip route add default via 192.168.1.1

# 6. Connect host-side veth to bridge (Flannel/Calico) or ENI (AWS VPC CNI)
ip link set veth0 master cni0
💡
The pause container trick: Every Pod starts with a hidden pause container (also called the "infra container"). Its only job is to hold the network namespace open. All other containers in the Pod join this namespace. If the app container crashes and restarts, the network namespace (and IP) is preserved.
03

CNI Plugins Compared

theory

There are many CNI plugins. Each makes different trade-offs between simplicity, performance, and features.

🟠 Flannel

Simplest

The simplest CNI plugin. Gives every node a subnet and wraps pod traffic in VXLAN tunnels (by default) to route between nodes. No Network Policy support. Great for learning but rarely used in production.

  • Overlay network using VXLAN (default) or UDP
  • No Network Policy support
  • Very easy to install — one kubectl apply
  • Best for: dev/test clusters, learning Kubernetes networking

🦁 Calico

Most Popular Production Choice

The gold standard for production. Can work without overlay (BGP, pure IP routing) which means near-native performance. Full Network Policy support including its own extended CiliumNetworkPolicy. Used by many cloud providers.

  • Can use BGP routing (no overlay, full line-rate performance)
  • Full Kubernetes NetworkPolicy + extended GlobalNetworkPolicy
  • Works in both overlay (VXLAN/IPIP) and non-overlay modes
  • Best for: on-prem, bare metal, EKS (with Calico for policy only)

🕸 Weave Net

Easy + Policy

Creates a mesh overlay network. Simpler to operate than Calico, supports Network Policy, includes built-in encryption. Slower than Calico in high-throughput scenarios due to the mesh overlay.

  • Overlay mesh — no BGP configuration needed
  • Supports Kubernetes NetworkPolicy
  • Optional built-in encryption (overhead: ~17%)
  • Best for: small to mid-size clusters needing simplicity + policy

🔶 AWS VPC CNI

Default on EKS

AWS-native CNI. Does NOT use an overlay. Instead, assigns actual VPC IP addresses directly to pods from ENIs (Elastic Network Interfaces). Pods are first-class VPC citizens — reachable from any VPC resource without NAT. This is what your EKS cluster is running.

  • No overlay — pods get real VPC IPs from ENIs
  • Native VPC routing — no tunneling overhead
  • Pod traffic visible in VPC Flow Logs, Security Groups apply
  • IP limit: tied to EC2 instance type's ENI/IP limits
  • Best for: all EKS clusters — it is the default and recommended choice

🐝 Cilium

eBPF-powered (Modern)

Next-generation CNI using eBPF (extended Berkeley Packet Filter) in the Linux kernel instead of iptables. Dramatically better performance and observability. Can completely replace kube-proxy. The fastest-growing CNI in production.

  • eBPF-based — bypasses iptables entirely for packet processing
  • Hubble: built-in deep observability and network flow visibility
  • L7 (HTTP/gRPC-aware) network policies
  • Best for: high-performance production clusters, modern cloud-native stacks

Step 4 — Understand IP limits per node

AWS VPC CNI has a hard limit on how many pods can run per node, based on the EC2 instance type's ENI capacity.

bash
# Check max pods per node (annotation set by aws-node)
kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.allocatable.pods}{"\n"}{end}'

# See current pod count per node
kubectl get pods -A -o wide --field-selector spec.nodeName=<node-name> | wc -l
⚠️
Formula: Max pods = (Number of ENIs × (IPs per ENI − 1)) + 2. For t3.small: 3 ENIs × (4 IPs − 1) + 2 = 11 pods max. For production workloads, this limit is a key reason to choose larger instance types or enable prefix delegation.

Pod-to-Pod Networking Deep Dive

hands-on ~10 min

Prove that pods can talk to each other across nodes — and trace the exact packet path.

Step 1 — Deploy two pods on different nodes
yaml — pod-a.yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-a
  labels:
    app: pod-a
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/hostname
            operator: In
            values: ["<NODE-1-NAME>"]  # replace with first node name
  containers:
  - name: netshoot
    image: nicolaka/netshoot
    command: ["sleep", "3600"]
yaml — pod-b.yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-b
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/hostname
            operator: In
            values: ["<NODE-2-NAME>"]  # replace with second node name
  containers:
  - name: nginx
    image: nginx
bash
# Get node names first
kubectl get nodes

# Apply both pods (after filling in node names above)
kubectl apply -f pod-a.yaml
kubectl apply -f pod-b.yaml

# Verify they are on different nodes
kubectl get pods -o wide
Step 2 — Test cross-node pod communication
bash
# Get pod-b's IP
POD_B_IP=$(kubectl get pod pod-b -o jsonpath='{.status.podIP}')
echo "pod-b IP: $POD_B_IP"

# Exec into pod-a and ping pod-b (cross-node!)
kubectl exec -it pod-a -- ping -c 4 $POD_B_IP

# Curl nginx on pod-b from pod-a
kubectl exec -it pod-a -- curl -s $POD_B_IP | head -5
✓ Expected — Cross-node ping works!
PING 192.168.42.67 (192.168.42.67): 56 data bytes
64 bytes from 192.168.42.67: seq=0 ttl=253 time=0.412 ms
64 bytes from 192.168.42.67: seq=1 ttl=253 time=0.389 ms

This proves pod-to-pod cross-node networking works with NO NAT.
TTL=253 means 3 hops: pod → node1 → VPC → node2 → pod
Step 3 — Inspect routing inside a pod
bash
# Look at pod-a's network interfaces
kubectl exec pod-a -- ip addr

# Look at pod-a's routing table
kubectl exec pod-a -- ip route

# Trace route from pod-a to pod-b
kubectl exec pod-a -- traceroute $POD_B_IP
✓ What you'll see inside pod-a
ip addr:
  eth0: 192.168.11.45/32   <-- /32 — single IP, AWS VPC style

ip route:
  default via 169.254.1.1 dev eth0     <-- link-local gateway
  169.254.1.1 dev eth0 scope link      <-- ARP proxy on node

traceroute to 192.168.42.67:
  1  169.254.1.1    <-- node's ARP proxy (veth gateway)
  2  192.168.42.67  <-- pod-b directly via VPC routing!
🔍
The 169.254.1.1 trick: AWS VPC CNI uses a link-local address (169.254.1.1) as a proxy ARP gateway inside the pod. When pod-a wants to send traffic to any destination, it sends to 169.254.1.1, and the host-side veth responds to ARP with its own MAC. The host then routes the traffic through the VPC.

Kubernetes Network Policies

theory + concept

By default, all pods can talk to all pods. Network Policies are Kubernetes's firewall rules at the pod level.

ANALOGYOffice Building Access Cards

Without Network Policies, the Kubernetes cluster is like an open-plan office — anyone can walk to any desk and talk to anyone.

Network Policies are like access card zones. You can say: "Only people from the HR team can enter the Finance department. Contractors can only access the lobby."

In Kubernetes terms: "Only pods with label app=frontend can connect to pods with label app=database on port 5432."

Network Policy Key Concepts

// podSelector

Which pods this policy applies TO (the target). Uses label selectors. Empty selector {} means "all pods in this namespace".

// policyTypes

Whether the policy controls Ingress (incoming), Egress (outgoing), or both. If you specify Ingress, only that direction is controlled.

// ingress / egress rules

A whitelist of allowed traffic. If a NetworkPolicy selects a pod, ALL traffic not explicitly allowed is denied. No policy = allow all.

// namespaceSelector

Allow traffic from specific namespaces (e.g., only from the monitoring namespace). Combines with podSelector for fine-grained control.

⚠️
Critical: Network Policies are additive whitelists. As soon as any NetworkPolicy selects a pod, all traffic to/from that pod is denied by default. You must explicitly allow what you want. A common mistake is creating a policy that accidentally blocks all egress (including DNS!), breaking the pod.

Install Calico for Network Policy on EKS

hands-on ~10 min

AWS VPC CNI handles pod IP assignment but has no built-in Network Policy enforcement. We add Calico as the policy engine (not as the CNI — just the policy controller).

💡
How this works: We keep AWS VPC CNI for pod networking (IP allocation, routing). We add Calico in "policy-only" mode. Calico installs iptables rules on each node to enforce NetworkPolicy objects. Two components, working together.
Step 1 — Install Calico operator
bash
# Install the Calico operator from Tigera
kubectl create -f https://raw.githubusercontent.com/projectcalico/calico/v3.27.0/manifests/tigera-operator.yaml

# Wait for the operator to be ready
kubectl rollout status deployment tigera-operator -n tigera-operator
Step 2 — Apply Calico Installation config for EKS

Create the Installation resource telling Calico to work in policy-only mode (no CNI, just policy enforcement).

yaml — calico-eks.yaml
apiVersion: operator.tigera.io/v1
kind: Installation
metadata:
  name: default
spec:
  cni:
    type: AmazonVPC      # Keep AWS VPC CNI for IP allocation
  calicoNetwork:
    bgp: Disabled         # No BGP needed — AWS VPC handles routing
---
apiVersion: operator.tigera.io/v1
kind: APIServer
metadata:
  name: default
spec: {}
bash
kubectl apply -f calico-eks.yaml

# Watch Calico components come up
kubectl get pods -n calico-system -w
✓ Expected — Calico pods running
NAME                                       READY   STATUS
calico-kube-controllers-xxx                1/1     Running
calico-node-aaaaa                          1/1     Running   <-- DaemonSet, one per node
calico-node-bbbbb                          1/1     Running
calico-typha-xxx                           1/1     Running
Step 3 — Verify Calico is enforcing policies

Hands-On Network Policy Lab

hands-on ~20 min

Build a 3-tier app (frontend → backend → database) and use NetworkPolicy to enforce that only the correct tiers can communicate.

// Lab Architecture

frontend (nginx) — can reach backend on port 8080
backend (httpd) — can reach database on port 5432
database (postgres) — only accepts connections from backend
Rule: frontend CANNOT reach database directly.

Step 1 — Create the app namespace and deployments
bash
# Create a dedicated namespace
kubectl create namespace app-demo

# Deploy frontend (nginx)
kubectl run frontend --image=nginx --labels="app=frontend,tier=web" -n app-demo

# Deploy backend (httpd acting as our app server)
kubectl run backend --image=httpd --labels="app=backend,tier=app" -n app-demo

# Deploy database (postgres)
kubectl run database \
  --image=postgres:15 \
  --labels="app=database,tier=db" \
  --env="POSTGRES_PASSWORD=secret" \
  -n app-demo

kubectl get pods -n app-demo -o wide
Step 2 — Verify all pods can talk before applying policies (baseline)
bash
DB_IP=$(kubectl get pod database -n app-demo -o jsonpath='{.status.podIP}')
echo "DB IP: $DB_IP"

# Try reaching database from frontend — should WORK before policy
kubectl exec -n app-demo frontend -- curl -s --connect-timeout 3 $DB_IP:5432 \
  && echo "✅ Connected (no policy yet)" || echo "❌ Blocked"
✓ Before policy — connection succeeds (default allow-all)
✅ Connected (no policy yet)   <-- This is the security gap we'll fix
Step 3 — Apply the default-deny policy to the namespace

First, lock down all traffic. Then explicitly open only what's needed.

yaml — default-deny.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: app-demo
spec:
  podSelector: {}          # {} = select ALL pods in this namespace
  policyTypes:
  - Ingress
  - Egress
  # No ingress or egress rules = deny everything
bash
kubectl apply -f default-deny.yaml

# Verify database is now unreachable from frontend
kubectl exec -n app-demo frontend -- curl -s --connect-timeout 3 $DB_IP:5432 \
  && echo "✅ Connected" || echo "❌ Blocked — policy works!"
✓ After deny-all — database is now blocked from frontend
❌ Blocked — policy works!
Step 4 — Allow DNS egress (or all pods break)

Deny-all blocks DNS too. Allow it explicitly so pods can resolve names.

yaml — allow-dns.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns-egress
  namespace: app-demo
spec:
  podSelector: {}             # all pods in namespace
  policyTypes: [Egress]
  egress:
  - ports:
    - protocol: UDP
      port: 53                # DNS
    - protocol: TCP
      port: 53
bash
kubectl apply -f allow-dns.yaml
Step 5 — Allow backend → database on port 5432
yaml — allow-backend-to-db.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-backend-to-database
  namespace: app-demo
spec:
  podSelector:
    matchLabels:
      app: database          # This policy protects the database pod
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: backend        # ONLY backend pods can connect
    ports:
    - protocol: TCP
      port: 5432
bash
kubectl apply -f allow-backend-to-db.yaml

# backend CAN now reach database
DB_IP=$(kubectl get pod database -n app-demo -o jsonpath='{.status.podIP}')
kubectl exec -n app-demo backend -- curl -s --connect-timeout 3 $DB_IP:5432 \
  && echo "✅ backend can reach database" || echo "❌ Blocked"

# frontend STILL CANNOT reach database — policy enforced!
kubectl exec -n app-demo frontend -- curl -s --connect-timeout 3 $DB_IP:5432 \
  && echo "✅ Connected (BAD!)" || echo "❌ Blocked — frontend correctly denied!"
✓ Final Verification — Zero-trust between tiers works
✅ backend can reach database
❌ Blocked — frontend correctly denied!

Your 3-tier network isolation is now working correctly.
Step 6 — Inspect all NetworkPolicies
bash
# List all policies in namespace
kubectl get networkpolicy -n app-demo

# Describe a specific policy
kubectl describe networkpolicy allow-backend-to-database -n app-demo

Troubleshooting CNI Issues

reference

Common CNI and networking problems you will encounter — and exactly how to diagnose and fix them.

Problem: Pod stuck in ContainerCreating — CNI not working

The most common CNI issue. Pod is scheduled but never gets an IP.

bash — Diagnosis
# Step 1: See the exact error
kubectl describe pod <pod-name> | grep -A 10 Events

# Step 2: Check kubelet logs on the node
kubectl get events --field-selector involvedObject.name=<pod-name> --sort-by='.lastTimestamp'

# Step 3: Check aws-node pod logs on that node
NODE=$(kubectl get pod <pod-name> -o jsonpath='{.spec.nodeName}')
kubectl logs -n kube-system -l k8s-app=aws-node --field-selector spec.nodeName=$NODE

# Step 4: Check if CNI binaries exist on the node
# (using a privileged pod on that node)
ls /opt/cni/bin/
ls /etc/cni/net.d/
🔧
Common causes: aws-node DaemonSet pod is crashing (check its logs), ENI IP exhaustion (node has no more IPs to give), missing IAM permissions for the node's role (needs AmazonEKS_CNI_Policy), or CNI config file missing/malformed.
Problem: Pod can't resolve DNS (name lookup fails)
bash — Diagnosis
# Test DNS resolution from inside a pod
kubectl exec -it <pod> -- nslookup kubernetes.default

# Check CoreDNS pods are running
kubectl get pods -n kube-system -l k8s-app=kube-dns

# Check CoreDNS logs for errors
kubectl logs -n kube-system -l k8s-app=kube-dns

# Check resolv.conf inside the pod
kubectl exec -it <pod> -- cat /etc/resolv.conf
💡
Quick fix: If you applied a default-deny NetworkPolicy, you must also add a rule allowing UDP/TCP port 53 egress to the kube-system namespace (where CoreDNS lives). This is the #1 mistake when implementing deny-all policies.
Problem: Pod-to-pod connectivity broken across nodes
bash — Diagnosis
# Test connectivity step by step
# 1. Can pods on the SAME node talk?
kubectl exec pod-a -- ping -c 2 $SAME_NODE_POD_IP

# 2. Can pods on DIFFERENT nodes talk?
kubectl exec pod-a -- ping -c 2 $DIFF_NODE_POD_IP

# 3. Check VPC route tables in AWS console — each node's subnet
#    must have routes for the pod CIDR
aws ec2 describe-route-tables \
  --filters "Name=tag:kubernetes.io/cluster/my-eks-cluster,Values=owned" \
  --region ap-south-1

# 4. Check Security Groups — node SGs must allow all traffic
#    between nodes (TCP/UDP all ports within the cluster SG)
aws ec2 describe-security-groups \
  --filters "Name=tag:aws:eks:cluster-name,Values=my-eks-cluster"
Problem: Debugging a NetworkPolicy — is it blocking?
bash — Diagnosis
# Check which policies apply to a pod
kubectl get networkpolicy -n <namespace> -o yaml

# Check iptables rules that Calico installed (on the node)
# (run inside a privileged pod on that node)
iptables -L -n | grep cali

# Use netshoot pod for comprehensive network debugging
kubectl run netshoot --rm -it --image=nicolaka/netshoot -n <namespace> -- bash

# Inside netshoot — test specific port connectivity
nc -zv <target-ip> <port>     # TCP port test
curl -v --connect-timeout 3 <target-ip>:<port>
